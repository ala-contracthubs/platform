# `poa/poa-case-workflow`

The heart of the POA module and its aggregate root (PoaCase). Declares + enforces the normative Sub-status transition table (17 working sub-statuses + #4a + 5 Cancelled terminals T1-T5; illegal (from,event) pairs return 400) — this POA sub-status table lives HERE in poa-case-workflow/domain (SubStatusTransitionValidator), not in the kernel; the kernel supplies only a context-free state-machine mechanism. AGGREGATE BOUNDARY: PoaCase references the backbone Case by CaseId (id reference, never object reference) and is authoritative for Sub-status, POA Type (immutable after documents_submitted), Principals (1-5) with Principal Authority Mode and per-Principal verification gate at #1, Attorneys (cap 5) with Attorney Authority Mode, Company-as-data parties + Authorised Signatory, the N-of-N approval gate at #8, Principal-Only Action enforcement (Approve Draft #8, Sign in Video Session #13 - non-delegable per Principal; payment delegable), wrong-type cancellation (T3), markers, and the per-user humanised status + next-action label. It DRIVES backbone Global Status and the one-Partner (Notary office) assignment ONLY by requesting transitions through the case-lifecycle port (BackboneCasePort) — it never writes the backbone Global Status, Partner slot, or platform audit log directly. Value objects: PoaType, PoaSubStatusKey, PrincipalAuthorityMode (joint|several|joint_and_several), AttorneyAuthorityMode, PartyType, CompanyPartyData, PassportNotarisationFlag, PrincipalOnlyAction, ApprovalTracker(N-of-N), OriginatingPrincipalRef, HumanisedStatus, NextActionLabel. Plugs into case-lifecycle only via the Manifest workflow contract.
